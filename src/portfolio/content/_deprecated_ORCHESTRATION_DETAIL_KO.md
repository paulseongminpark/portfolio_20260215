## 3) 시작 — 세션을 열면 아무것도 없다

### SectionTitle

- eyebrow: 시작
- title: 세션을 열면 아무것도 없다


어제 같은 말을 세 번 했다. 커밋하기 전에 상태 파일을 갱신해라. push를 빼먹지 마라. 프로젝트 허브를 업데이트해라. CLAUDE.md에 규칙으로 써놨다. 74줄. Claude는 절반도 읽지 않았다.

다음 날 아침, 세션을 열었다. Claude는 어제를 모른다. 세 번의 지적도, 그 전 세션에서 내린 결정도, 어떤 파일을 왜 그렇게 수정했는지도. 프로젝트가 하나일 때는 다시 설명하면 됐다. 6개를 동시에 돌리면 매 세션마다 복구에 30분, 작업에 30분.

그래서 더 만들기 시작했다. 에이전트를 만들고, 규칙을 쓰고, 팀을 짰다. 문제가 생기면 에이전트를 하나 더 만들었다. 규칙이 부족하면 줄을 더 썼다.

## 3) 시작 — 기억 × 거버넌스

### SectionTitle

- eyebrow: 시작
- title: 기억 × 거버넌스


모든 사람이 "AI가 기억을 못 한다"고 말한다. 절반만 맞다.

세션이 끝나면 결정, 실패, 패턴이 사라진다. 이건 기억의 문제다. 그런데 기억만으로는 안 된다. CLAUDE.md에 규칙을 74줄 써놨는데, Claude는 절반도 읽지 않았다. 헌법이지만 사법부가 없는 헌법이었다. 이건 기억이 아니라 거버넌스의 문제다.

기억만 있으면 "맥락은 있지만 규칙을 안 지키는 AI"가 된다. 거버넌스만 있으면 "규칙은 지키지만 맥락이 없는 AI"가 된다. 이 두 가지를 동시에 풀어야 했다. 기억은 mcp-memory가, 거버넌스는 Hook이 맡게 된다. 하지만 그 답에 도착하기까지 6번의 재설계가 필요했다.

## 5) 설계 2막 — 42K

### SectionTitle

- eyebrow: 설계 · 2막
- title: 42K


세션을 열었다. 일을 시작하기도 전에 42K 토큰이 사라져 있었다.

200K 예산의 20%. CLAUDE.md를 로드하고, rules/를 로드하고, 에이전트 24개의 정의를 로드하고, MEMORY.md를 로드한다. 그게 42K다. 남은 158K로 실제 작업을 해야 하는데, 100K를 넘으면 compact가 시작된다. 에이전트를 하나 추가할 때마다 baseline이 올라간다. 기능이 늘어날수록 작업 공간이 줄어든다.

CLAUDE.md에 규칙을 74줄 썼다. Claude는 절반도 읽지 않았다. 규칙을 더 쓸수록 Claude는 더 혼란스러워했다. 문서로 행동을 바꿀 수 없었다.

시스템이 만든 사람의 이해 범위를 넘어섰다.

"잘 통합됐는지 검증이 어렵고, 이전 잔재가 쌓여있고, 역할 겹침이 두렵고, 과도하지 않은지 두렵다. 단순하면서 완벽하게 작동하는 걸 못 할까봐 두렵다. 내가 다 이해하지 못하고 있는 기분." 이것이 v5 설계를 시작하기 직전에 내가 한 말이다.

**[다이어그램 2 — 42K 토큰 소비 분해: CLAUDE.md ~12K + rules ~8K + agents ~15K + MEMORY.md ~5K + skills ~2K = ~42K. 200K 전체 대비 20%. 오른쪽에 "일하기 전에 사라진 예산".]**

## 6) 설계 3막 — What만 남기다

### SectionTitle

- eyebrow: 설계 · 3막
- title: What만 남기다


CLAUDE.md 한 줄을 바꿨다. 6개 프로젝트가 동시에 반응했다. 에이전트 하나를 수정했더니 3개 팀의 체인이 깨졌다.

이건 설정 파일이 아니었다. 이건 커널이었다.

CLAUDE.md는 시스템 콜 인터페이스다. rules/는 커널 모듈이다. 에이전트는 프로세스다. Hook은 인터럽트 핸들러다. "OS라 부르지만 실체는 스크립트 모음"이었다. 스크립트 모음을 진짜 OS로 만들어야 했다.

"빼는 것이 강화다."

24개를 3개로 줄였다. code-reviewer, compressor, commit-writer. 격리 실행이 필요한 것만 남겼다 — 자기 코드를 자기가 리뷰하면 편향이 생기고, 압축은 다른 모델이 해야 객관적이고, 커밋 메시지에 Opus는 과하다. 나머지 21개는 Skills로 흡수하거나 Claude가 직접 수행한다.

CLAUDE.md 74줄을 38줄로. baseline 42K에서 28K로. 33% 절감. 기능은 줄지 않았다.

커널은 What만 가진다. How는 프로세스에 위임한다. CLAUDE.md에는 "무엇"만 남기고, "어떻게"는 Skills와 Workers에 넘겼다.

3기둥. Guards — 인터럽트 핸들러. Claude가 실수하기 전에 환경이 잡는다. 차단이 아니라 경고. 차단하면 Claude가 우회한다. Skills — 유저 프로그램. 호출될 때만 존재하고, 비용은 0. Workers — 격리 프로세스. 분리된 눈이 필요한 작업만.

Claude를 똑똑하게 만들려 하지 않았다. 환경을 똑똑하게 만들었다. 규칙이 단순해야 Codex도 Gemini도 읽는다.

**[다이어그램 3 — 3기둥 OS 아키텍처: PAUL(방향+판단+승인) → CLAUDE/Opus 1M(설계+실행+조율) → Guards(14 Hooks, 인터럽트 핸들러) / Skills(13, 유저 프로그램) / Workers(3, 격리 프로세스). OS 레이어 시각화.]**

**[다이어그램 4 — Before/After: 왼쪽 v3.3(에이전트 24, 스킬 14, CLAUDE.md 74줄, baseline ~42K) → 오른쪽 v6(Workers 3, Skills 13, Hooks 14, baseline ~15K). -87.5% 에이전트, -48.6% CLAUDE.md, -64% baseline.]**

## 7) 설계 4막 — AI 없이 측정한다

### SectionTitle

- eyebrow: 설계 · 4막
- title: AI 없이 측정한다


v5는 잘 돌아가는 기계였다. 그런데 눈이 없고, 입이 없고, 세션끼리 귀가 없었다.

"지금 건강한가?" — 이 질문에 v5는 답할 수 없었다. 도구는 사용자가 들여다봐야 상태를 안다. 살아있는 시스템은 스스로 안다. v6의 출발점은 여기였다.

measure.py. AI를 사용하지 않는다. 훅이 작동하는지. MCP 서버가 살아있는지. CLAUDE.md와 현실이 일치하는지. 파이프라인이 멈춰있진 않은지. 수백 개 항목을 물리적으로 확인한다. true 아니면 false.

AI가 자기 자신을 평가하면 편향이 생긴다. "규칙을 잘 지키고 있는가?"라고 Claude에게 물으면 높은 확률로 "네"라고 답한다. 체온계는 환자에게 열이 있냐고 묻지 않는다. 직접 잰다. measure.py는 시스템의 체온계다.

측정 결과는 루프로 들어간다. 측정 → 진단 → 제안 → Paul 승인 → 적용 → 재측정. 이 순환이 돌면, 잊혀도 시스템이 다시 발견한다. "나중에 고쳐야지"는 실패한다. 기억에 의존하는 개선은 확장되지 않는다.

자동화의 경계는 명시적이다. 측정은 자동이지만, 수정은 승인이 필요하다. 시스템이 스스로 판단해서 경계를 넓히면 신뢰가 무너진다. 사람이 대신 써주는 건 v5다. 스스로 쓰는 게 v6다. 그러나 스스로 판단하는 건 아직 아니다. 그 경계는 Paul이 정한다.

**[다이어그램 5 — Auto-iterate 루프: measure → diagnose → propose → [PAUL 승인] → apply → verify → measure. 6단계 순환. Paul이 개입하는 지점만 다른 색. "측정은 AI 없이. 승인은 사람이."]**

## 9) 강화 — 규율의 원천

### SectionTitle

- eyebrow: 강화
- title: 규율의 원천


세 번의 전환이 있었다. 매번 규율의 원천이 달랐다.

v4, Context as Currency. 200K 토큰 한계가 규율을 만들었다. 에이전트를 더 추가하면 baseline이 올라가고, baseline이 올라가면 작업 공간이 줄어든다. 물리적 제약이 "빼라"고 강제했다. 제약에서 오는 규율.

v5, The Machine. 아키텍처가 규율을 만들었다. Guards가 실수를 잡고, Skills가 절차를 보장하고, Workers가 격리를 유지한다. 규칙이 문서에 있는 것이 아니라 코드로 실행된다. 사람이 기억하지 않아도 Hook이 강제한다. 설계에서 오는 규율.

v6, The Living System. 자기인식이 규율을 만들었다. 시스템이 자기 건강을 측정하고, 문제를 발견하고, 개선을 제안한다. 규율의 원천이 외부(제약)에서 구조(설계)로, 구조에서 내부(자기인식)로 이동했다.

규율이 외부에서 내부로 이동한다. 제약은 없어질 수 있다. 설계는 깨질 수 있다. 자기인식은 깨져도 다시 발견한다.

**[다이어그램 7 — 규율의 원천 진화: v4 Context as Currency(제약 — 토큰 한계) → v5 The Machine(설계 — 아키텍처) → v6 The Living System(자기인식 — 측정 루프). 수평 타임라인, 화살표, 각 노드에 규율의 원천.]**

## 9) 강화 — 이 원리는 도구를 넘는다

### SectionTitle

- eyebrow: 강화
- title: 이 원리는 도구를 넘는다


이 시스템에서 발견한 원리 다섯 가지. Claude Code가 내일 사라져도 유효하다.

{{b:맥락의 경제학.}} 올바른 3K 토큰이 잘못된 15K를 이긴다. 이건 토큰 예산의 문제이면서, 동시에 모든 조직의 정보 과부하 문제다. 회의에서 30페이지를 다 읽는 것보다 핵심 3페이지를 정확히 읽는 게 의사결정 품질을 높인다.

{{b:실패 우선.}} mcp-memory에서 Failure 노드가 recall 최우선이다. compact 시 성공은 버려도 실패는 남긴다. 포스트모템이 성공 사례집보다 가치 높다. 어떤 조직의 지식 관리에서도 동일하다.

{{b:환경 기반 규율.}} Claude를 교육하는 대신 환경을 바꿨다. Hook이 행동을 유도한다. 이건 nudge architecture — 사용자를 교육하지 않고 인터페이스를 바꿔서 행동을 유도하는 UX 설계의 핵심 원리와 같다.

{{b:단일 쓰기 권한.}} 여러 AI가 같은 파일을 수정하면 충돌한다. Claude만 Write, 나머지는 Read. 분산 시스템의 CQRS 패턴이고, 조직에서 "의사결정 권한을 명확히 하라"는 거버넌스 원칙이다.

{{b:자기참조적 원칙.}} foundation/ 문서가 세션에 주입되어 행동을 규정하고, 행동의 결과로 문서가 갱신된다. 원칙이 행동을 만들고, 행동이 원칙을 바꾼다. 문화가 조직을 만들고, 조직이 문화를 바꾸는 것과 같다.

가고 싶은 지점은 더 만드는 것이었다. 더 연결하고, 더 자동화하고, 더 정교하게. 가야만 하는 지점은 정돈하는 것이었다. 깎고, 구조화하고, 환경에 맡기는 것. 이 둘이 수렴하는 데 6번의 재설계가 필요했다.

이 포트폴리오 페이지 자체가 mcp-memory의 노드에서 추출된 것이다. 시스템이 자기 자신을 설명하는 순환. 여기서부터가 시작이다.
