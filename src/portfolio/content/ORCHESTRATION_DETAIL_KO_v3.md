## 3) 시작 — 세션을 열면 아무것도 없다

### SectionTitle

- eyebrow: 시작
- title: 세션을 열면 아무것도 없다


어제 같은 말을 세 번 했다. 커밋하기 전에 상태 파일을 갱신해라. push를 빼먹지 마라. 프로젝트 허브를 업데이트해라. 규칙은 이미 CLAUDE.md에 적어뒀다. 74줄이었다. 그런데 Claude는 그 절반도 제대로 반영하지 못했다.

다음 날 아침 세션을 열면, 전날의 판단이 이어지지 않았다. 세 번의 지적도, 그 전 세션에서 내린 결정도, 어떤 파일을 왜 그렇게 수정했는지도. 프로젝트가 하나일 때는 감당할 수 있었다. 하지만 여섯 개를 동시에 돌리기 시작하자, 세션을 복구하는 데 30분, 실제 작업에 30분이 드는 날이 반복됐다.

이 문제를 줄이려면 판단과 절차를 시스템 안에 고정해야 했다. 그래서 에이전트를 만들고, 규칙을 정리하고, 팀을 나누기 시작했다.

## 3) 시작 — 기억 × 거버넌스

### SectionTitle

- eyebrow: 시작
- title: 기억 × 거버넌스


처음에는 이 문제를 단순히 기억의 부재로 봤다. 세션이 끝나면 결정, 실패, 패턴이 사라지고, 다음 세션은 늘 처음부터 다시 시작됐다. 분명 기억의 문제였다.

하지만 기억만 복구된다고 해결되지는 않았다. 규칙을 문서에 적어둬도 Claude는 일관되게 따르지 않았고, 프로젝트마다 같은 실수가 반복됐다. 맥락을 보존하는 것과 규칙을 실제로 집행하는 것은 다른 문제였다.

필요했던 것은 둘 중 하나가 아니었다.
이전 판단을 이어받으면서도, 정해둔 절차를 실제 동작으로 강제하는 구조가 필요했다.

## 6) 설계 3막 — What만 남기다

### SectionTitle

- eyebrow: 설계 · 3막
- title: What만 남기다


전환점은 작은 수정에서 왔다. CLAUDE.md 한 줄을 바꿨는데 6개 프로젝트가 동시에 반응했고, 에이전트 하나를 수정했더니 3개 팀의 체인이 깨졌다.

그때부터 CLAUDE.md 는 시스템 전반에서 에이전트 동작의 경계를 정하는 기준에 더 가까웠다.

그래서 구조를 다시 나눴다. CLAUDE.md에는 방향과 기준만 남기고, 실제 실행 방법은 다른 레이어로 내렸다. 커널에는 What만 두고, How는 실행 레이어로 넘기는 방식이다.

원칙은 하나였다. 빼는 것이 강화다.

이 과정에서 24개 에이전트를 3개로 줄였다. code-reviewer, compressor, commit-writer만 남겼다. 기준은 명확했다. 정말로 격리된 실행이 필요한가. 그렇지 않으면 Skills로 흡수하거나 Claude가 직접 수행하게 했다. 그 결과 CLAUDE.md는 74줄에서 38줄로 줄었고, baseline은 42K에서 28K까지 내려갔다.

핵심은 기능 삭제가 아니라 재배치였다. 커널은 What만 가지고, How는 실행 레이어에 넘긴다. Guards는 먼저 개입해 실수를 막고, Skills는 필요할 때만 호출되며, Workers는 별도 시선이나 격리가 필요한 작업만 맡는다.

14개 Hook은 하나의 설정 파일에서 돌아간다. 규칙은 문서보다 코드일 때 강제된다. Codex, Gemini, Claude가 같은 규칙을 읽도록 .rulesync/에 원본을 하나 두고 각 AI의 설정 파일을 자동 생성하게 만들었다.

```json .claude/settings.json — Hook Gates
{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Edit|Write",
      "hooks": [{
        "type": "command",
        "command": "if STATE.md|CLAUDE.md|CHANGELOG.md → '⚠️ 중요 파일 변경. /sync 권장'"
      }]
    }],
    "Stop": [
      { "command": "copy-session-log.py" },
      { "command": "save_session()" },
      { "command": "if STATE.md uncommitted → exit 1" }
    ]
  }
}
```

이후부터 초점은 모델을 더 똑똑하게 만드는 데 있지 않았다. 실수하기 쉬운 지점을 구조로 막고, 반복되는 판단을 환경에 고정하는 쪽으로 설계 방향이 바뀌었다.

**[다이어그램 3 — 3기둥 OS 아키텍처: 나(방향+판단+승인) → CLAUDE/Opus 1M(설계+실행+조율) → Guards(14 Hooks, 인터럽트 핸들러) / Skills(13, 유저 프로그램) / Workers(3, 격리 프로세스). OS 레이어 시각화.]**

**[다이어그램 4 — Before/After: 왼쪽 v3.3(에이전트 24, 스킬 14, CLAUDE.md 74줄, baseline ~42K) → 오른쪽 v6(Workers 3, Skills 13, Hooks 14, baseline ~15K). -87.5% 에이전트, -48.6% CLAUDE.md, -64% baseline.]**

## 7) 설계 4막 — AI 없이 측정한다

### SectionTitle

- eyebrow: 설계 · 4막
- title: AI 없이 측정한다


3막까지의 재설계로 실행 구조는 정리됐다. Guards가 실수를 잡고, Skills가 절차를 실행하고, Workers가 격리된 작업을 맡는다. 이 시스템을 v5, The Machine이라 불렀다. 실행 구조로는 충분히 작동했다. 다만 자기 상태를 측정하는 기준이 없었고, 세션 사이 변화도 안정적으로 이어받지 못했다.

“지금 건강한가?” 이 질문에 답하려면, 먼저 상태를 직접 측정해야 했다. 그래서 measure.py를 만들었다.

여기에는 AI를 쓰지 않는다. 훅이 실제로 작동하는지, MCP 서버가 살아 있는지, CLAUDE.md의 규칙과 현재 상태가 어긋나지 않았는지, 파이프라인이 멈춰 있지 않은지를 직접 확인한다. 수백 개 항목을 기계적으로 검사하고, 결과는 true 아니면 false다.

```python auto-iterate/src/measure.py
# L1: Infrastructure
check_claude_md_agents()         # 에이전트 파일 존재
check_security()                 # .env, 키 파일 노출 스캔
check_external()                 # GitHub API, 포트폴리오 사이트 상태

# L2: Config Integrity
check_claude_md_projects()       # CLAUDE.md ↔ 실제 폴더 정합
check_rules_sync()               # pipeline-rules.md ↔ phase-rules.json

# L3: Knowledge
check_orphan_nodes()             # 연결 없는 노드 비율
check_broken_edges()             # 삭제된 노드를 가리키는 엣지
check_session_freshness()        # 마지막 세션 저장 경과일

# L4: Pipeline
check_living_docs()              # STATE.md + CHANGELOG.md 존재 + 최신성
check_pipeline_freshness()       # ACTIVE 파이프라인 정체 여부

# L5: Growth
check_test_suites()              # pytest + validate_pipeline.py 통과
check_weekly_delta()             # 7일 전 점수 대비 변화

# → score = 100 * (passed / total)
```

AI가 자기 상태를 스스로 평가하게 두면 편향이 들어갈 수 있다. 그래서 이 단계는 모델의 해석이 아니라 검증 가능한 체크로 처리했다.

측정 결과는 바로 루프로 들어간다. 측정, 진단, 제안, 사람의 승인, 적용, 재측정. 이 순환이 있어야 잊혀도 다시 발견할 수 있다.

세션 종료를 처리하는 /session-end도 같은 맥락에서 만들었다. 세션이 끝날 때 compressor가 대화를 압축하고, PDR(Post-Dialogue Review)이 전체 대화를 분석해 관찰, 결정, 실패, 인사이트로 분류한다. 분류된 기록은 mcp-memory에 저장되고, 상태 파일이 갱신되고, 커밋이 만들어진다. 다음 세션에서 recall()이 호출되면, 어제의 결정과 실패가 오늘의 맥락으로 다시 들어온다.

여기서부터 v6의 방향이 분명해졌다.
시스템이 먼저 상태를 읽고, 문제를 드러내고, 다음 조치를 제안한다. 다만 적용 범위와 승인 경계는 여전히 사람이 정한다.

**[다이어그램 5 — Auto-iterate 루프: measure → diagnose → propose → [승인] → apply → verify → measure. 6단계 순환. 내가 개입하는 지점만 다른 색. "측정은 AI 없이. 승인은 사람이."]**

## 8) 실체

### SectionTitle

- eyebrow: 실체
- title: 커널


앞선 설계는 각각의 기능을 따로 만든 과정이었다.
실제로 중요한 변화는 이 조각들이 하나의 운영 루프로 연결되기 시작했다는 데 있었다.
중심에는 orchestration이 있다. 여기서 규칙과 실행 경계를 정의한다.

mcp-memory는 세션 밖의 결정, 실패, 패턴을 보존한다. context-cascade는 그중 지금 필요한 것만 골라 현재 세션에 주입한다. auto-iterate는 상태를 측정하고 개선 루프를 제안한다. index-system은 프로젝트 간 연결을 추적한다.

context-export는 시스템 맥락을 다섯 축으로 구조화한다. 정체성, 방법론, 상태, 결정, 에코시스템이 그 축이다. 이 구조를 다른 AI와 연동하면(Gemini, Perplexity, ChatGPT), Claude Code에서 진행된 시스템 변경사항, 최근 생성된 노드, 온톨로지 연결을 다른 모델도 읽을 수 있다. 필요하면 DB에 직접 접근해 조회하는 방식까지 포함해 설계했다.

이 구조의 바닥에는 foundation/이 있다. 철학, 원칙, 워크플로우를 담은 레이어로, 세션이 열릴 때마다 주입되어 행동 기준으로 작동한다.

정리하면 이 시스템은 규칙, 기억, 주입, 측정을 분리해서 두는 대신 하나의 루프로 묶었다. 그 결과 여러 프로젝트를 동시에 돌려도 세션마다 처음부터 다시 설명하지 않아도 되는 구조에 가까워졌다.

**[다이어그램 6 — 커널 에코시스템: 중심에 orchestration(커널). 주변에 CLAUDE.md+rules(시스템 콜) / Guards(인터럽트 핸들러) / Skills(유저 프로그램) / Workers(격리 프로세스) / mcp-memory(파일 시스템) / context-cascade(메모리 매니저) / auto-iterate(시스템 모니터) / index-system(의존성 그래프). 연결선 + OS 역할 라벨.]**

## 9) 강화 — 규율의 원천

### SectionTitle

- eyebrow: 강화
- title: 규율의 원천


이 구조는 세 번의 전환을 거치며 달라졌다.
각 단계마다 시스템을 통제하는 방식도 달랐다.

v4, Context as Currency.
처음의 규율은 토큰 한계에서 왔다. 200K 예산 안에서 baseline이 커질수록 작업 공간은 줄어든다. 에이전트를 더 붙이면 바로 비용으로 되돌아왔다. 이 시기의 규율은 외부 제약에서 만들어졌다.

v5, The Machine.
다음 단계에서는 설계가 규율의 역할을 맡았다. Guards가 실수를 잡고, Skills가 절차를 보장하고, Workers가 격리를 유지한다. 규칙을 사람이 기억하는 대신, 구조가 먼저 개입하도록 바꿨다.

v6, The Living System.
지금 단계에서 규율은 측정 루프에서 나온다. 시스템이 자기 상태를 점검하고, 문제를 발견하고, 개선안을 제안한다. 사람은 승인과 범위 결정을 맡고, 시스템은 상태 파악과 반복 점검을 맡는다.

이 변화는 단순한 버전 업이 아니었다.
처음에는 외부 제약이 시스템을 통제했고, 다음에는 설계가 그 역할을 맡았고, 지금은 측정과 피드백 루프가 그 자리를 대신한다.

**[다이어그램 7 — 규율의 원천 진화: v4 Context as Currency(제약 — 토큰 한계) → v5 The Machine(설계 — 아키텍처) → v6 The Living System(자기인식 — 측정 루프). 수평 타임라인, 화살표, 각 노드에 규율의 원천.]**

## 9) 강화 — 이 원리는 도구를 넘는다

### SectionTitle

- title: 이 원리는 도구를 넘는다


이 시스템을 만들면서 남은 원리는 다섯 가지다. 다른 도구로 옮겨가도 그대로 적용할 수 있다.

{{b:맥락의 경제학.}} 많다고 좋은 정보는 아니다. 지금 필요한 맥락을 정확히 읽는 편이 더 중요하다.

{{b:실패 우선.}} 성공보다 실패가 먼저 남아야 같은 실수를 줄일 수 있다.

{{b:환경 기반 규율.}} 모델에게 더 많이 설명하는 대신, 실수하기 쉬운 지점에 Hook과 검증 절차를 배치했다. 행동을 바꾸는 가장 안정적인 방법은 설명보다 환경 설계에 가까웠다.

{{b:단일 쓰기 권한.}} 여러 AI가 같은 파일을 동시에 수정하게 두면 충돌이 생긴다. 그래서 최종 쓰기 권한은 Claude에만 두고, 다른 모델은 읽기·추출·검증 역할에 머물게 했다.

{{b:자기참조적 원칙.}} foundation 문서는 세션 안에서 행동 기준으로 주입되고, 그 행동의 결과는 다시 문서에 반영된다. 원칙이 행동을 만들고, 행동의 결과가 다시 원칙을 갱신한다.

가고 싶은 지점은 더 만드는 것이었다. 더 연결하고, 더 자동화하고, 더 정교하게. 가야만 하는 지점은 정돈하는 것이었다. 깎고, 구조화하고, 환경에 맡기는 것. 이 둘이 수렴하는 데 여러 번의 재설계가 필요했다.

이 글 역시 mcp-memory에 남은 노드와 기록을 바탕으로 다시 구성한 것이다. 중요한 점은 하나다. 이제 이 시스템은 세션마다 처음부터 설명해야 하는 대상이 아니라, 이전 판단을 이어받고 다음 조치를 준비하는 운영 구조에 가까워졌다.
