## 3) 시작 — 세션을 열면 아무것도 없다

### SectionTitle

- eyebrow: 시작
- title: 세션을 열면 아무것도 없다


어제 같은 말을 세 번 했다. 커밋하기 전에 상태 파일을 갱신해라. push를 빼먹지 마라. 프로젝트 허브를 업데이트해라. 규칙은 이미 CLAUDE.md에 적어뒀다. 74줄이었다. 그런데 Claude는 그 절반도 제대로 반영하지 못했다.

다음 날 아침, 세션을 열었다. Claude는 어제를 모른다. 세 번의 지적도, 그 전 세션에서 내린 결정도, 어떤 파일을 왜 그렇게 수정했는지도. 프로젝트가 하나일 때는 다시 설명하면 됐다. 하지만 6개를 동시에 돌리기 시작하자, 매 세션마다 복구에 30분, 실제 작업에 30분이 들었다.

반복 설명을 줄이려면 판단과 절차를 시스템 안에 고정해야 했다. 그래서 더 만들기 시작했다. 에이전트를 만들고, 규칙을 쓰고, 팀을 짰다.

## 3) 시작 — 기억 × 거버넌스

### SectionTitle

- eyebrow: 시작
- title: 기억 × 거버넌스


많은 사람은 이 문제를 기억의 부재로 설명한다. 맞는 말이다. 하지만 충분하지 않다.

세션이 끝나면 결정, 실패, 패턴이 사라진다. 이건 기억의 문제다. 그런데 기억만으로는 충분하지 않았다. 규칙을 문서에 써놔도 Claude는 따르지 않았다. 헌법은 있었지만, 집행 구조는 없었다. 이건 거버넌스의 문제이기도 했다.

기억만 있으면 맥락은 있지만 규칙을 지키지 않는 시스템이 된다. 거버넌스만 있으면 규칙은 있지만 맥락이 없다. 내가 만들려던 것은 둘 중 하나가 아니라, 둘을 함께 갖는 구조였다.

### SectionTitle

- eyebrow: 설계 · 1막

처음에는 역할을 더 잘게 나누면 문제가 줄어들 거라고 생각했다. 반복되는 판단을 시스템 안에 고정하면 세션 복구 비용도 줄어들 것처럼 보였다.

그래서 에이전트를 만들었다. 처음에는 14개였다. 곧 24개로 늘었다. 팀도 나눴다. ops, build, analyze, maintain.

에이전트 24개. 스킬 14개. 팀 4개. 외부 AI 3개. 그때는 구성이 커질수록 시스템도 강해진다고 믿었다.

## 5) 설계 2막 — 42K

### SectionTitle

- eyebrow: 설계 · 2막
- title: 42K


문제가 분명해진 건 세션을 다시 열었을 때였다. 일을 시작하기도 전에 42K 토큰이 사라져 있었다.

CLAUDE.md, rules/, 에이전트 정의, MEMORY.md를 읽는 데 42K가 들었다. 200K 예산의 20%였다. 기능을 더할수록 baseline은 올라갔고, 실제 작업 공간은 줄어들었다.

문제는 비용만이 아니었다. 구조도 무거워졌다. 규칙을 더 쓸수록 Claude는 더 혼란스러워했다. 문서만으로는 행동을 바꿀 수 없었다. 시스템이 복잡해질수록, 내가 만든 시스템을 내가 한눈에 설명하기 어려워졌다.

그때 처음으로 알았다. 이건 단순히 더 많은 기능의 문제가 아니었다. 구조를 다시 설계해야 하는 문제였다.

**[다이어그램 2 — 42K 토큰 소비 분해: CLAUDE.md ~12K + rules ~8K + agents ~15K + MEMORY.md ~5K + skills ~2K = ~42K. 200K 전체 대비 20%. 오른쪽에 "일하기 전에 사라진 예산".]**

## 6) 설계 3막 — What만 남기다

### SectionTitle

- eyebrow: 설계 · 3막
- title: What만 남기다


전환점은 작은 수정에서 왔다. CLAUDE.md 한 줄을 바꿨는데 6개 프로젝트가 동시에 반응했고, 에이전트 하나를 수정했더니 3개 팀의 체인이 깨졌다.

그때 알았다. 이건 설정 파일이 아니라 시스템 전체의 동작 경계를 정하는 커널에 가까웠다.

이후부터 구조를 운영체제처럼 다시 보기 시작했다. CLAUDE.md는 호출 규약, rules/는 모듈, 에이전트는 프로세스, Hook은 인터럽트 핸들러처럼 다뤘다. 하지만 실제로는 느슨하게 붙은 스크립트 묶음에 가까웠고, 그래서 그 묶음을 진짜 운영 구조로 다시 설계해야 했다.

원칙은 하나였다. 빼는 것이 강화다.

24개를 3개로 줄였다. code-reviewer, compressor, commit-writer만 남겼다. 기준은 단순했다. 격리된 실행이 정말 필요한가. 나머지는 Skills로 흡수하거나 Claude가 직접 수행하게 했다. CLAUDE.md는 74줄에서 38줄로, baseline은 42K에서 28K까지 내려갔다.

핵심은 기능을 없앤 게 아니라 배치를 다시 정한 것이었다. 커널은 What만 가지고, How는 실행 레이어에 넘긴다. Guards는 먼저 개입하고, Skills는 필요할 때만 호출되며, Workers는 다른 시선이 필요한 작업만 맡는다.

14개 Hook은 하나의 설정 파일에서 돌아간다. 규칙은 문서보다 코드일 때 강제된다. Codex, Gemini, Claude가 같은 규칙을 읽도록 .rulesync/에 원본을 하나 두고 각 AI의 설정 파일을 자동 생성하게 만들었다.

내가 하려던 건 Claude를 더 똑똑하게 만드는 일이 아니라, 환경을 더 똑똑하게 만드는 일이었다.

**[다이어그램 3 — 3기둥 OS 아키텍처: 나(방향+판단+승인) → CLAUDE/Opus 1M(설계+실행+조율) → Guards(14 Hooks, 인터럽트 핸들러) / Skills(13, 유저 프로그램) / Workers(3, 격리 프로세스). OS 레이어 시각화.]**

**[다이어그램 4 — Before/After: 왼쪽 v3.3(에이전트 24, 스킬 14, CLAUDE.md 74줄, baseline ~42K) → 오른쪽 v6(Workers 3, Skills 13, Hooks 14, baseline ~15K). -87.5% 에이전트, -48.6% CLAUDE.md, -64% baseline.]**

## 7) 설계 4막 — AI 없이 측정한다

### SectionTitle

- eyebrow: 설계 · 4막
- title: AI 없이 측정한다


3막까지의 설계로 구조는 잡혔다. Guards가 실수를 잡고, Skills가 절차를 실행하고, Workers가 격리된 작업을 맡는다. 이 시스템을 v5, The Machine이라 불렀다. 실행 구조로는 잘 돌아가는 기계였다. 하지만 자기 상태를 측정하는 눈이 없었고, 세션 사이 변화를 이어받는 구조도 부족했다.

“지금 건강한가?” 이 질문에 답하려면, 먼저 상태를 직접 재야 했다. 그래서 measure.py를 만들었다.

여기에는 AI를 쓰지 않는다. 훅이 실제로 작동하는지, MCP 서버가 살아 있는지, CLAUDE.md의 규칙과 현재 상태가 어긋나지 않았는지, 파이프라인이 멈춰 있지 않은지를 직접 확인한다. 수백 개 항목을 기계적으로 검사하고, 결과는 true 아니면 false다.

이유는 단순했다. AI가 자기 자신을 평가하면 편향이 생긴다. 체온계는 환자에게 열이 있냐고 묻지 않는다. 직접 잰다. measure.py는 이 시스템의 체온계다.

측정 결과는 바로 루프로 들어간다. 측정, 진단, 제안, 사람의 승인, 적용, 재측정. 이 순환이 있어야 잊혀도 다시 발견할 수 있다.

세션이 끝나면 모든 맥락이 사라진다 — 이건 시작에서 정의한 문제였다. /session-end는 이 문제의 직접적인 답이다. 세션이 끝나는 시점에 compressor가 대화를 압축하고, PDR(Post-Dialogue Review)이 전체 대화를 분석해서 관찰, 결정, 실패, 인사이트로 분류한다. 분류된 기록은 mcp-memory에 저장되고, 상태 파일이 갱신되고, 커밋이 만들어진다. 다음 세션에서 recall() — mcp-memory에서 관련 지식을 꺼내는 함수 — 이 호출되면, 어제의 결정과 실패가 오늘의 맥락으로 돌아온다. measure.py가 시스템 건강을 측정하고, /session-end가 세션 지식을 구조화한다.

v6는 시스템이 먼저 관찰하고 제안하는 구조다. 하지만 적용 범위는 아직 사람이 정한다.

**[다이어그램 5 — Auto-iterate 루프: measure → diagnose → propose → [승인] → apply → verify → measure. 6단계 순환. 내가 개입하는 지점만 다른 색. "측정은 AI 없이. 승인은 사람이."]**

## 8) 실체

### SectionTitle

- eyebrow: 실체
- title: 커널


이 시스템의 중심은 orchestration이다. 여기서 규칙과 실행 경계를 정의한다.

mcp-memory는 세션 밖의 결정, 실패, 패턴을 보존한다. context-cascade는 그중 지금 필요한 것만 골라 현재 세션에 주입한다. auto-iterate는 상태를 측정하고 개선 루프를 제안한다. index-system은 프로젝트 간 연결을 추적한다.

context-export는 시스템 맥락을 다섯 축으로 구조화한다 — 정체성, 방법론, 상태, 결정, 에코시스템. 이 구조가 다른 AI에 연동되면(Gemini, Perplexity, ChatGPT), Claude Code로 작업하는 모든 시스템 변경사항, 최근 생성된 노드, 온톨로지 연결을 파악할 수 있다. DB에도 직접 접근 가능하며, 쿼리 가능하게 설계했다.

이 구조의 바닥에는 foundation/이 있다 — 철학, 원칙, 워크플로우. 세션이 열릴 때마다 주입되어 행동 기준이 된다.

즉 규칙, 기억, 주입, 측정이 하나의 운영 루프로 묶인다. 이 구조 위에서 여러 프로젝트를 동시에 돌릴 수 있게 됐다.

**[다이어그램 6 — 커널 에코시스템: 중심에 orchestration(커널). 주변에 CLAUDE.md+rules(시스템 콜) / Guards(인터럽트 핸들러) / Skills(유저 프로그램) / Workers(격리 프로세스) / mcp-memory(파일 시스템) / context-cascade(메모리 매니저) / auto-iterate(시스템 모니터) / index-system(의존성 그래프). 연결선 + OS 역할 라벨.]**

## 9) 강화 — 규율의 원천

### SectionTitle

- eyebrow: 강화
- title: 규율의 원천


세 번의 전환이 있었다. 매번 규율의 원천이 달랐다.

v4, Context as Currency. 200K 토큰 한계가 규율을 만들었다. 에이전트를 더 추가하면 baseline이 올라가고, baseline이 올라가면 작업 공간이 줄어든다. 물리적 제약이 "빼라"고 강제했다. 제약에서 오는 규율.

v5, The Machine. 아키텍처가 규율을 만들었다. Guards가 실수를 잡고, Skills가 절차를 보장하고, Workers가 격리를 유지한다. 규칙이 문서에 있는 것이 아니라 코드로 실행된다. 사람이 기억하지 않아도 Hook이 강제한다. 설계에서 오는 규율.

v6, The Living System. 여기서 규율은 측정 루프에서 나온다. 시스템이 자기 상태를 점검하고, 문제를 발견하고, 개선안을 제안한다. 물론 적용은 아직 승인 경계 안에 있다. 그럼에도 규율의 원천은 분명히 이동했다. 외부 제약에서 구조 설계로, 구조 설계에서 자기 관찰로 옮겨갔다.

규율이 외부에서 내부로 이동한다. 제약은 없어질 수 있다. 설계는 깨질 수 있다. 자기인식은 깨져도 다시 발견한다.

**[다이어그램 7 — 규율의 원천 진화: v4 Context as Currency(제약 — 토큰 한계) → v5 The Machine(설계 — 아키텍처) → v6 The Living System(자기인식 — 측정 루프). 수평 타임라인, 화살표, 각 노드에 규율의 원천.]**

## 9) 강화 — 이 원리는 도구를 넘는다

### SectionTitle

- eyebrow: 강화
- title: 이 원리는 도구를 넘는다


이 시스템에서 발견한 원리 다섯 가지. Claude Code가 내일 사라져도 유효하다.

{{b:맥락의 경제학.}} 많다고 좋은 정보는 아니다. 지금 필요한 맥락을 정확히 읽는 편이 더 중요하다.

{{b:실패 우선.}} 성공보다 실패가 먼저 남아야 같은 실수를 반복하지 않는다.

{{b:환경 기반 규율.}} Claude에게 더 많이 설명하는 대신, 실수하기 쉬운 지점에 Hook을 배치했다. 환경으로 에이전트를 유도하는 쪽을 택했다.

{{b:단일 쓰기 권한.}} 여러 AI가 같은 파일을 동시에 수정하게 두면 충돌이 생긴다. 그래서 최종 쓰기 권한은 Claude에만 두고, 다른 모델은 읽기·추출·검증 역할에 머물게 했다.

{{b:자기참조적 원칙.}} foundation 문서는 세션 안에서 행동 기준으로 주입되고, 그 행동의 결과는 다시 문서에 반영된다. 원칙이 행동을 만들고, 행동의 결과가 다시 원칙을 갱신한다.

가고 싶은 지점은 더 만드는 것이었다. 더 연결하고, 더 자동화하고, 더 정교하게. 가야만 하는 지점은 정돈하는 것이었다. 깎고, 구조화하고, 환경에 맡기는 것. 이 둘이 수렴하는 데 여러 번의 재설계가 필요했다.

이 페이지도 mcp-memory에 남은 노드와 기록에서 다시 구성한 것이다. 시스템이 자기 자신을 설명할 수 있게 되었을 때, orchestration은 도구를 넘어 구조가 된다.
