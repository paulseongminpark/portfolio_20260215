## 2-a — 좌측 고정 앵커 인덱스 {#2-a}

**레이아웃:** 좌 220px 고정 사이드바(앵커 링크) + 우측 스크롤 영역

**배치 선택 이유:**

- **긴 페이지의 방향감 문제 해결**: 단일 페이지 포트폴리오에서 사용자가 "지금 어디 있지?"를 느낄 수 있음. 고정 인덱스가 이를 방지.
- **앵커 하이라이트**: 스크롤 위치에 따라 활성 섹션이 강조됨 → 사용자 현재 위치를 자동 표시.
- **법률/컨설팅 문서 레이아웃에서 차용**: 긴 문서를 읽을 때 좌측 목차가 있으면 신뢰감 형성.
- **단점**: 모바일에서 좌측 사이드바를 처리하는 방식이 복잡해질 수 있음.

**WorkDetail**: 좌측 OVERVIEW / PROBLEM / SOLUTION 앵커 인덱스로 동일 패턴 유지

## 2-b — 풀스크린 스냅 스크롤 {#2-b}

**레이아웃:** 각 섹션이 100vh. 스냅 스크롤로 한 번에 하나씩 전환. `01 / 05` 카운터 표시.

**배치 선택 이유:**

- **섹션을 명함처럼 제시**: 각 섹션이 독립적인 슬라이드. 방문자가 "넘기는" 경험을 하게 됨.
- **프레젠테이션 감각**: 포트폴리오를 피치덱처럼 제시. 짧은 시간에 보여줘야 할 때 효과적.
- **카운터**: `01 / 05`로 전체 분량을 미리 알려줌 → 이탈 방지.
- **단점**: 섹션 내 콘텐츠가 많으면 100vh 안에 다 담기 어려움. 텍스트 길이 통제 필요.

**WorkDetail**: 각 상세 섹션도 100vh 스냅 적용 (01 / 03 카운터)

## 2-c — 타이포그래피 크기 대비 전면 {#2-c}

**레이아웃:** 컬러나 선 없이 폰트 크기 대비(96px ↔ 14px)만으로 위계 생성.

**배치 선택 이유:**

- **타이포그래피가 디자인임을 보여줌**: 레이아웃 요소 없이 크기와 굵기의 대비만으로 시선을 이끄는 것이 가장 어려운 디자인 도전.
- **강렬한 첫인상**: 큰 제목이 화면을 채울 때의 시각적 충격 → 기억에 남음.
- **미니멀리즘의 극단**: 요소를 줄이다 보면 결국 타이포만 남는다는 미학적 실험.
- **단점**: 콘텐츠 양이 많아질수록 레이아웃 일관성 유지가 어려움.

**WorkDetail**: 프로젝트 제목을 화면 절반 크기로. 소제목은 극소형.

## 2-d — 카드 그리드 + 고정 헤더 {#2-d}

**레이아웃:** 상단 고정 헤더(이름 + 섹션 링크) + 본문은 카드 그리드.

**배치 선택 이유:**

- **SaaS 대시보드 감각**: 프로젝트를 카드로 제시하면 "관리되는 작업물" 인상. Notion, Linear 스타일.
- **스캐너빌리티 최대화**: 카드는 각각 독립적 단위 → 관심 있는 것만 선택적으로 볼 수 있음.
- **고정 헤더**: 스크롤 중에도 항상 접근 가능한 네비게이션. 실제 웹앱과 동일한 패턴.
- **단점**: 카드 사이즈를 통일하면 내용 길이가 다른 항목 처리가 어색할 수 있음.

**WorkDetail**: 3컬럼 카드 그리드로 OVERVIEW / PROBLEM / SOLUTION 배치

## 2-e — 좌검정 고정 / 우흰 스크롤 스플릿 {#2-e}

**레이아웃:** 좌 50% = `#0a0a0a` 고정 (이름/역할/연락처) + 우 50% = `#ffffff` 스크롤 (섹션 내용)

**배치 선택 이유:**

- **아이덴티티를 항상 노출**: 왼쪽에 이름과 역할이 고정되어 있어 어디를 스크롤해도 "누구의 포트폴리오인지" 명확.
- **흑백 대비의 극대화**: 화면 자체가 흑/백 두 개의 영역으로 나뉘어져 page-02의 흑백 미학을 공간으로 확장.
- **인터뷰/프레젠테이션 시나리오**: 면접관에게 노트북을 보여줄 때, 좌측 고정 정보가 항상 보임.
- **단점**: 좌측 정보가 단순해서 여백이 많아 보일 수 있음. 좌측 콘텐츠를 어떻게 채울지 고민 필요.

**WorkDetail**: 좌검정 고정 (프로젝트 제목/메타) + 우흰 스크롤 (상세 내용)
