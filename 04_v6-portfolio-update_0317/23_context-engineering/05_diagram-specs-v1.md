# Context Engineering 상세 페이지 — 다이어그램 최종 스펙

> 대상: 설계(1~5) + 실체(6) + 강화(7) = 총 7개
> 스타일 통일: Anthropic 블로그 미학. 크림/화이트 배경, 최소 선, 깔끔한 타이포. C4 Container 레벨. 장식 없음.
> 온톨로지 페이지와 같은 시각 언어. 두 페이지를 나란히 열었을 때 한 시스템에서 나온 것으로 보여야 함.

---

## [다이어그램 1] 만들수록 줄어들었다

**위치**: 설계 1막 끝. "끊기는 곳이 문제였다" 직후.
**형식**: C4 스타일. 가로 타임라인 + 면적 대비.
**목적**: 도구가 늘어날수록 생각할 공간이 줄어드는 역설을 한 장으로. 숫자("24→15→3")는 텍스트에서 이미 말했으니, 다이어그램은 **공간의 압축감**을 보여준다. 독자가 "만들수록 좁아졌구나"를 시각적으로 느끼게.

**내용**:
- 가로축 3개 스냅샷: v3.0 / v3.3.1 / v5.0
- 각 스냅샷은 하나의 직사각형(전체 컨텍스트 윈도우).
  - v3.0 (200K): 전체가 좁다. 그 안에서 "시스템 규칙 42K"(진한 영역) + "도구 정의 24개"(중간 영역) = 합치면 거의 전부. "작업 공간"(밝은 영역)이 아주 얇은 띠.
  - v3.3.1 (200K): 같은 전체 너비. "도구 정의 15개"로 줄어듦. 작업 공간이 눈에 띄게 커짐.
  - v5.0 (1M): 전체가 5배로 넓어짐. "도구 정의 3개"는 거의 안 보일 정도. 작업 공간이 압도적으로 큼.
- v3.0과 v5.0의 면적 대비가 핵심. 도구를 줄인 것 + 창이 커진 것, 둘 다 보여야 한다.
- 각 스냅샷 아래에 작은 글씨: "24 tools" / "15 tools" / "3 tools"
- 스냅샷 사이에 화살표 없음. 나란히 놓는 것만으로 충분.

**스타일**: 온톨로지 다이어그램 1(타입 팽창)과 같은 무게. 숫자가 주인공. 색은 3개만 — 시스템 규칙(고정, 진한), 도구 정의(줄어드는, 중간), 작업 공간(늘어나는, 밝은). 여백 넉넉.

**캡션**: "만들수록 생각할 공간이 줄었다. 줄일수록 늘었다."

## [다이어그램 2] 4개의 층

**위치**: 설계 2막 끝. "나누는 것 자체가 썩는 것을 막는 구조였다" 직후.
**형식**: C4 스타일. 수직 4개 레이어 스택.
**목적**: 정보의 수명이 다르면 사는 곳도 달라야 한다는 핵심 통찰을 구조화. 온톨로지 다이어그램 4(6레이어 성숙 지도)와 같은 위치 — 이 페이지의 뼈대.

**내용**:
- 아래(L0)에서 위(L3)로 4개 레이어. 아래가 가장 안정적, 위로 갈수록 휘발.
- 각 레이어:
  - **L0 Static** (~15K): "규칙, 설정, 도구 정의. 모든 세션에서 동일."
    - 구성요소 작은 박스: CLAUDE.md, rules/, MEMORY.md
    - 오른쪽 라벨: "수명: 영구. 변경: 드묾."
  - **L1 Session** (~200 tokens): "어제의 결정, 미해결 질문. 세션마다 한 번."
    - 구성요소: get_context(), recall()
    - 오른쪽 라벨: "수명: 세션. 변경: 매번."
    - 숫자 강조: "200토큰으로 전체 맥락 복구 (기존 1,700 대비)"
  - **L2 Task** (가변): "규모 추정 → 읽기 전략. 작업 시작 시."
    - 구성요소: Gate 판단, Progressive Reading
    - 오른쪽 라벨: "수명: 작업. 변경: 작업마다."
  - **L3 Live** (연속): "무거워지면 대응. 작업 내내."
    - 구성요소: auto_remember, Compact, 재주입
    - 오른쪽 라벨: "수명: 순간. 변경: 연속."
- L0과 L1 사이에 점선 경계: "Prompt Caching 경계 — 여기 아래가 안정적이어야 캐시가 동작한다"
- 왼쪽에 수직 화살표: "안정적 ← → 휘발"

**스타일**: 온톨로지 다이어그램 4(6레이어)와 시각적으로 대응. 같은 수직 스택, 같은 레이어 표현. 두 페이지를 나란히 놓으면 "기억의 레이어"와 "사고의 레이어"가 대구를 이루는 것이 보여야 한다. L0이 가장 넓고 진한 색, L3이 가장 얇고 밝은 색 — 안정성이 면적으로 드러남.

**캡션**: "수명이 다른 정보는 다른 곳에 산다."

## [다이어그램 3] 규모에 따라 전략이 갈린다

**위치**: 설계 3막, "아주 많으면 분산시킨다" 직후. 3막 다이어그램 3개 중 첫 번째.
**형식**: C4 스타일. 수평 분기 플로우.
**목적**: "파일을 열기 전에 먼저 물어야 할 것이 있었다"의 시각적 답. 규모라는 하나의 숫자가 전략 전체를 결정한다는 단순함.

**내용**:
- 상단: 진입점 하나. "관련 소스가 얼마나 되는가?"
- 아래로 3개 분기:
  - **≤300K → Gate A "직접"**
    - Opus 아이콘 하나. "전부 직접 읽는다."
    - 하위: "목차 → 이름 → 서명 → 본문" (다이어그램 4 미리보기, 작게)
    - 라벨: "대부분의 일상 작업"
  - **300K~800K → Gate B "정찰"**
    - Codex 아이콘 → Opus 아이콘. "보조가 정찰, 주력이 판단."
    - 하위: "파일 목록 → 중요도 순위 → 상위만 읽기"
    - 라벨: "큰 프로젝트"
  - **800K+ → Gate C "분산"**
    - 여러 아이콘(Codex, Gemini, Local) → Opus. "나눠 읽고, 모아서 추론."
    - 하위: "추출 계획 → 병렬 실행 → 갭 판단"
    - 라벨: "전체 코드베이스"
- 3개 분기의 너비/높이가 오른쪽으로 갈수록 커짐 — 규모가 커질수록 구조가 복잡해진다는 것이 면적으로 드러남.

**스타일**: 3막은 "확신"의 감정. 깔끔하고 확정적인 느낌. 분기가 명확하고 선이 직선. 온톨로지 다이어그램 5(3중 검색)와 같은 위치 — "작동 메커니즘"을 보여주는 역할.

**캡션**: "열기 전에 먼저 묻는다. 얼마나 되는가?"

## [다이어그램 4] 읽기의 순서

**위치**: 설계 3막, 다이어그램 3 직후. 3개 중 두 번째.
**형식**: C4 스타일. 수직 깔때기(funnel). 1/2 너비 또는 전체 너비.
**목적**: "전문을 바로 읽지 않는다"의 시각적 증명. 위에서 아래로 갈수록 토큰이 늘어나지만 도달하는 빈도는 줄어든다. 대부분의 작업이 본문까지 가지 않는다는 것.

**내용**:
- 위(넓음, 밝음)에서 아래(좁음, 진함)로 5단계:
  - **1. 인덱스** (00_index.md) — ~100 tokens
    - "전체 구조를 한눈에"
  - **2. Phase 인덱스** — ~200 tokens
    - "해당 영역만"
  - **3. 이름 목록** (get_symbols_overview) — ~500 tokens
    - "함수/클래스 이름만, 본문 없이"
  - **4. 서명** (find_symbol, include_body=False) — ~200 tokens
    - "입출력만. 내부 로직은 아직."
  - **5. 본문** (Read) — 가변
    - "여기까지 오면 진짜 필요한 것"
- 각 단계 오른쪽에 토큰 수.
- 단계 1~4의 합이 ~1,000 토큰. 5단계(본문)만 가변.
- 깔때기 옆에 수직 화살표: "대부분 여기서 끝남" → 3~4단계를 가리킴.
- 하단에 작은 텍스트: "서명만 보면 되는데 전문을 읽는 건 낭비다."

**스타일**: 온톨로지 다이어그램 6(승격 경로)과 같은 수직 흐름이되, 방향이 반대 — 승격은 아래→위(성장), 읽기는 위→아래(깊어짐). 깔때기가 좁아지면서 색이 진해짐.

**캡션**: "본문은 마지막이다."

## [다이어그램 5] 무거워지면

**위치**: 설계 3막, 다이어그램 4 직후. 3개 중 세 번째.
**형식**: C4 스타일. 수평 타임라인 + 임계점.
**목적**: 컨텍스트가 무거워질 때 단계적으로 대응하는 구조. "한 번에 전부 날리는 게 아니라, 단계별로, 안전하게"의 시각화. 핵심은 **압축 전에 3곳에 기록한다**는 안전장치.

**내용**:
- 가로축: 토큰 사용량. 왼쪽(0)에서 오른쪽(1M)으로.
- 3개의 임계점이 수직선으로 표시:
  - **~500K**: "저장"
    - 아래에: "checkpoint → mcp-memory에 중간 상태"
    - 라벨: 자율
  - **~700K**: "압축"
    - 아래에: "compact 필수. 압축 전 3곳 기록:"
    - 3개 박스 (작게): "인덱스 갱신" / "mcp-memory save_session()" / "작업 일지"
    - 라벨: 필수
  - **~900K**: "넘긴다"
    - 아래에: "세션 교체. /session-end 실행."
    - 라벨: 강력 권장
- 가로축 위에 색 그라데이션: 왼쪽(밝음, 여유) → 오른쪽(진함, 위험).
- 700K 임계점의 "3곳 기록" 부분이 시각적으로 강조 — 이것이 "잃는 것이 없다"의 근거.

**스타일**: 3막의 마지막. 확신의 마무리. 깔끔하고 정보 밀도 높음. 온톨로지 다이어그램에는 없는 유형(타임라인+임계점)이라 CE 페이지만의 고유한 시각 자산.

**캡션**: "기록하고 나서 압축하면 잃는 것이 없다."

### 3막 레이아웃 노트

- 다이어그램 3, 4, 5는 **순서대로 배치**. 가로 3연타가 아니라 수직 순서.
  - 3(Gate 분기)이 "어떻게 읽을지"를 정하고
  - 4(Progressive Reading)가 "어떤 순서로 읽을지"를 보여주고
  - 5(Compact)가 "읽다가 무거워지면 어떻게 하는지"를 닫는다.
- 온톨로지 4막(BEAT 2)이 가로 3연타 몽타주였다면, CE 3막은 **수직 3단 시퀀스**. 다른 리듬.
- 3막 텍스트가 밀도 있으니, 다이어그램도 텍스트 사이사이에 끼워 넣기보다 텍스트 뒤에 한꺼번에 배치하는 것이 호흡에 맞을 수 있음. 최종 레이아웃에서 판단.

## [다이어그램 6] 에코시스템

**위치**: 실체 끝. "남은 것. 규칙 파일에 3줄." 직후.
**형식**: C4 Container 레벨. 전체 너비.
**목적**: CE가 독립 시스템이 아니라 이미 존재하는 6개 시스템을 관통하는 원칙이라는 것을 한 장으로. 온톨로지 다이어그램 8(삼자 순환)이 "세 존재의 관계"를 보여줬다면, 이것은 "하나의 원칙이 여섯 곳을 꿰뚫는 것"을 보여준다.

**내용**:
- 중심에 "Context Engineering" (원칙). 작은 박스, 강조색.
  - 안에: "무엇을 보여줄 것인가?"
- 중심에서 6개 방향으로 방사:
  - **01 orchestration** (조율): "규칙, 스킬, 상태 관리. CE 규칙의 집."
  - **08 documentation-system** (구조): "폴더 구조 = 읽기 순서. phase-rules.json."
  - **09 context-cascade** (선별): "4-Layer, Gate, Compact. CE의 본체."
  - **10 index-system** (지도): "25개 프로젝트, 38.7M 토큰. 규모 추정의 근거."
  - **06 mcp-memory** (지속): "get_context 200토큰. 세션 간 다리."
  - **11 user-guide** (운영): "포스트잇, 세션 라이프사이클. 사람 쪽 인터페이스."
- 6개 노드 사이에도 연결선:
  - 08→09: "구조가 선별을 결정"
  - 10→09: "규모가 Gate를 결정"
  - 06→01: "기억이 조율에 사용"
  - 09→06: "Compact가 기억을 저장"
- 하단에 숫자: "92 files · 6 systems · 1 principle"

**스타일**: 방사형이지만 장식적이지 않게. 연결선은 직선, 레이블은 작게. 중심("무엇을 보여줄 것인가?")이 가장 강조. 나머지 6개는 같은 크기, 같은 무게. "하나의 원칙이 전부를 관통한다"는 느낌. 온톨로지 다이어그램 8(삼자 순환)과 같은 전체 너비, 같은 무게감.

**캡션**: "6개 시스템. 하나의 원칙."

## [다이어그램 7] 저장과 제시의 순환

**위치**: 강화 맨 끝. CE 페이지 전체의 최종 다이어그램. "그 다음 질문이 있다." 직후.
**형식**: C4 스타일. 순환 다이어그램. 전체 너비.
**목적**: CE 페이지의 마지막 이미지. mcp-memory 페이지와 CE 페이지를 하나로 잇는 다리. 온톨로지 다이어그램 12(Three Layers)가 "업계 지도에서 내 위치"였다면, 이것은 "두 프로젝트가 하나의 순환을 이룬다"는 것. 두 페이지를 읽은 독자가 "아, 이게 하나구나"를 느끼는 순간.

**내용**:
- 좌우 대칭 순환:
  - **왼쪽: mcp-memory** (저장)
    - "기억의 구조를 설계한다"
    - 하위: "타입 · 관계 · 성숙 · 감쇠"
    - 아이콘 또는 심볼: 그래프 노드 형태
  - **오른쪽: Context Engineering** (제시)
    - "사고의 환경을 설계한다"
    - 하위: "레이어 · 선별 · 읽기 · 압축"
    - 아이콘 또는 심볼: 레이어 스택 형태
  - **위쪽 화살표** (mcp-memory → CE):
    - "꺼낸다 — 적절한 시점에, 적절한 양만큼"
    - recall(), get_context()
  - **아래쪽 화살표** (CE → mcp-memory):
    - "저장한다 — 구조화해서, 관계와 함께"
    - save_session(), auto_remember
  - **중심**: "추론" — 두 화살표가 만나는 곳.
    - "그 위에서 생각한다"
- 하단, 순환 바깥:
  - 세 줄 질문 (작은 글씨, 왼쪽→오른쪽):
    - "어떻게 물어볼까?" → **Prompt Engineering**
    - "무엇을 보여줄까?" → **Context Engineering**
    - "무엇을 구조화할까?" → **?**
  - 세 번째 질문의 "?"가 mcp-memory 페이지를 가리킴. (시각적으로 — 화살표 또는 열린 끝.)

**스타일**: CE 페이지의 마지막이자, 두 페이지를 잇는 다리. 온톨로지 다이어그램 12(Three Layers)와 같은 무게감이되, 닫지 않고 열어둠. "?"가 있어야 한다. 색은 두 페이지의 색을 모두 사용 — mcp-memory 쪽은 온톨로지 페이지의 악센트 색, CE 쪽은 CE 페이지의 악센트 색. 중심("추론")은 두 색의 중간이거나 별도 강조.

**캡션**: 없음. 마지막 다이어그램은 캡션 없이 여백으로 닫는다. 텍스트("그 다음 질문이 있다.")가 이미 캡션 역할.

# 온톨로지 페이지와의 시각적 대응 노트

두 페이지는 같은 포트폴리오의 두 축이다. 시각적으로도 대구를 이뤄야 한다:

- **설계의 뼈대**: 온톨로지 = 6레이어 성숙 지도 (수직 스택). CE = 4개의 층 (수직 스택). 같은 형식, 다른 내용.
- **작동 메커니즘**: 온톨로지 = 3중 검색 + 승격 + 캡처 (가로 3연타). CE = Gate + 읽기 + Compact (수직 3단). 다른 리듬.
- **전체 그림**: 온톨로지 = 삼자 순환 (Paul × Claude × 온톨로지). CE = 에코시스템 (1 원칙 × 6 시스템). 같은 무게, 다른 구조.
- **마지막 다이어그램**: 온톨로지 = Three Layers (업계 지도). CE = 저장과 제시의 순환 (두 페이지의 다리). 온톨로지가 "밖을 향해 열리고", CE가 "안으로 순환을 닫는다".
